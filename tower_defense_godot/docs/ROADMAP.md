# Roadmap

本文整理 `Path Bender Tower Defense` 後續開發方向。Roadmap 不是承諾表，而是幫助排序：哪些事情先做會讓後續更穩，哪些可以等核心體驗更完整後再做。

## 目前狀態

目前版本：`Beta 0.7.4`

已完成：

- 主選單、規則、排行榜、難度選擇。
- 45x25 建造區，塔佔地 2x2。
- 地面敵人可 8 方向移動。
- 飛行敵人可直飛終點。
- 50 波結束後有攻擊量審查員。
- 塔與敵人數值資料化。
- `main.gd` 已拆出 UI、渲染、路徑、波次、存檔、戰鬥、建造、音效、對話視窗。
- Web 版支援 Netlify GitHub 自動部署。
- Web 中文字型修正與字型瘦身。
- 後期效能已加入分層繪製與高負載低畫質模式。
- 手機橫式介面已壓低上方資訊欄，並加入雙指縮放與塔周圍操作按鈕。

## 近期目標

### 1. 穩定版本整理

目標：讓目前 `Beta 0.6.x` 可以成為可持續更新的穩定基礎。

建議工作：

- 整理目前未 commit 的文件與程式變更。
- 確認 README、GAME_LOGIC、CHANGELOG、ROADMAP 都一致。
- 本地跑 Godot headless 檢查。
- 測試普通難度第 1、20、40、50 波。
- 測試 Netlify 自動部署後中文字與效能。

完成標準：

- 本地可執行。
- Web 可開啟。
- 中文正常。
- 第 40 波後仍可操作。
- 文件不再出現舊版 `Beta 0.3.1`、`30x20`、Boss 秒數排行榜等過期資訊。

### 2. 視覺精修第一批

目標：在不大幅增加效能負擔的前提下，讓塔與敵人更容易辨識。

建議工作：

- 塔外觀建立一致風格。
- Boss 外觀與名稱更明確對應。
- 敵人受擊效果保留，但高負載可降級。
- 子彈外觀簡化成可讀但低成本。
- 重要提示如生命損失、波次提醒、退款飄字保持清楚。

完成標準：

- 玩家可以一眼分辨砲塔、箭塔、冰凍塔。
- 玩家可以一眼分辨螞蟻、獨角仙、蟑螂、蜘蛛、蝗蟲、Boss。
- 第 40 波後不因視覺精修明顯變卡。

### 3. 平衡調整第二批

目標：讓普通與困難模式後期仍有壓力，但不要變成純數值碾壓。

建議工作：

- 重新檢查第 20 波後擊殺獎勵。
- 重新檢查第 30、40 波敵人數量與堆疊。
- 調整獨角仙、蜘蛛、蝗蟲在後期的角色。
- 檢查困難模式是否可通關但需要策略。
- 檢查箭塔低成本迷宮是否太強。

完成標準：

- 普通難度可通關，但第 40 波後仍可能漏怪。
- 困難難度可通關，但需要更好的建塔路線與升級順序。
- 第 50 波結束時玩家不應累積過多無法花掉的金幣。

### 4. 儲存與排行榜整理

目標：讓存檔與排行榜更可信，避免後續調整版本時資料格式混亂。

建議工作：

- 為 save/meta 增加版本欄位。
- 文件化 save JSON 格式。
- 排行榜只保留目前需要的攻擊量與耗時。
- 若資料格式變更，提供簡單相容或重置策略。

完成標準：

- 舊資料不會讓遊戲壞掉。
- 排行榜顯示與實際記錄一致。
- 文件可說明資料存在哪裡、格式長什麼樣。

## 下一階段

### 5. 波次資料化

目標：把出怪節奏、波次組成、Boss 規則逐步資料化。

目前 `wave_manager.gd` 已經集中波次邏輯，但若之後要大量設計關卡，建議新增：

- `data/waves.json`
- 或 `data/difficulties.json`

可資料化內容：

- 每波敵人種類比例。
- 每波數量倍率。
- 同時生成上限。
- Boss 波設定。
- 飛行敵人出現頻率。

完成標準：

- 調整出怪不用大改 `wave_manager.gd`。
- 新增關卡或模式可以透過資料表完成大部分設定。

### 6. 測試關卡與效能基準

目標：建立固定測試場景，比較改動前後是否變慢。

建議工作：

- 建立一個固定滿圖塔測試狀態。
- 建立一個第 40 波大量敵人測試狀態。
- 記錄 Chrome 的 CPU、LCP、互動流暢度。
- 未來每次視覺改動都用相同場景比較。

完成標準：

- 可重現「後期大量子彈」壓力場景。
- 可判斷某次改動是否讓效能變差。

### 7. 畫質選項

目標：讓玩家可手動選擇畫質，低階設備也能玩。

建議選項：

- 高畫質：完整子彈、血條、衝擊波。
- 標準：目前自動降級。
- 低畫質：常態減少子彈、血條與特效。

完成標準：

- 畫質切換不影響傷害計算。
- 低畫質在第 40 波後明顯更順。

### 8. 美術素材流程

目標：從幾何圖形逐步進入可替換的美術素材。

建議先做：

- 小型 sprite atlas。
- 統一敵人尺寸規則。
- 統一塔底座、砲管、升級標記規則。
- Web 圖片壓縮與快取規則。

完成標準：

- 新增一個敵人外觀不用大改 renderer。
- 新增一個塔外觀有固定格式。
- Web 載入大小仍可控。

## 長期方向

### 9. 多關卡與地圖

可能方向：

- 不同起點/終點。
- 多入口。
- 特殊地形。
- 固定障礙。
- 不同地圖尺寸。

前置條件：

- 波次資料化。
- 地圖資料化。
- 靜態層重繪流程更穩定。

### 10. 更多塔與敵人機制

可能方向：

- 毒塔。
- 電塔。
- 範圍增益塔。
- 裝甲敵人。
- 分裂敵人。
- 會破壞塔或暫時封鎖格子的敵人。

前置條件：

- `combat_manager.gd` 的狀態效果更資料化。
- 效能基準建立完成。

### 11. 操作體驗精修

可能方向：

- 更清楚的建造合法/非法提示。
- 塔詳細資訊面板。
- 升級路線預覽。
- 一鍵暫停。
- 波次預告。
- 手機版操作優化。

前置條件：

- UI layout 更模組化。
- 手機版 viewport 再測一次。

### 12. 發布品質

可能方向：

- 正式版版本號。
- 網站說明頁。
- 遊戲內 credits。
- 錯誤回報方式。
- 玩家資料隱私說明。

前置條件：

- 核心遊戲體驗穩定。
- Web 部署流程穩定。

## 暫緩項目

這些項目有價值，但現在做容易拖慢核心開發：

- 完整帳號系統。
- 線上排行榜後端。
- 多人模式。
- 大量圖片素材替換。
- 完整關卡編輯器。
- 手機 App 打包。

## 開發排序建議

建議順序：

1. 穩定目前 `Beta 0.6.x`。
2. 補測試與效能基準。
3. 視覺精修第一批。
4. 平衡調整第二批。
5. 波次資料化。
6. 畫質選項。
7. 美術素材流程。
8. 多地圖與更多機制。

這個順序的理由是：先讓現在的遊戲穩，接著建立比較基準，再開始加畫面與內容。這樣每次變漂亮時，都還能知道有沒有把遊戲變慢。
